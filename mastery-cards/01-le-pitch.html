DocuQuery AI

Mehdi Bargach · 2026

Un système de questions-réponses augmenté qui permet d'interroger en langage naturel n'importe quel document propriétaire (PDF, texte, tableur) et d'obtenir une réponse précise avec la page et le paragraphe exact d'où elle vient.

1

Le problème

Situation : tu as un document propriétaire — un PRD, un contrat, un rapport d'audit. Tu veux une info précise.

OptionCe qui se passePourquoi c'est cassé
Coller dans ChatGPT Le LLM a une fenêtre de contexte = quantité max de texte qu'il peut lire en une fois. GPT-4 ≈ 128K tokens, Claude ≈ 200K tokens. 1) Le LLM traite bien le début et la fin, mais rate l'info au milieu — c'est le « lost in the middle », documenté par la recherche.

2) Plus le texte est long, plus c'est cher en tokens.

3) Aucune citation — le LLM dit « le budget est 18M » mais quelle page ? Invérifiable.
Demander SANS le document Le LLM n'a jamais vu ton doc (propriétaire, pas dans ses données d'entraînement) Il hallucine = il génère une réponse crédible (bon ton, bonne structure) mais les faits sont inventés.
Ctrl+F Recherche par mots-clés exacts uniquement Tu tapes « budget » → rate « coût d'acquisition » ou « CAC » ou « $3,600 ». C'est une recherche lexicale (mots exacts) et pas sémantique (le sens de ta question).

Pourquoi le problème ne disparaît pas avec des context windows plus grands : même avec 1M+ tokens, RAG reste pertinent — la recherche ciblée donne de meilleures réponses que noyer le LLM dans 200 pages, ça coûte ~100x moins cher, et un context window géant ne donne toujours aucune citation vérifiable.

Le vrai besoin : interroger un document que l'IA n'a jamais vu, obtenir une réponse fondée, avec la source exacte pour vérifier.

2

RAG — comment ça marche

RAG = Retrieval-Augmented Generation (génération augmentée par la récupération d'information).

Un LLM est puissant pour comprendre et rédiger, mais il ne connaît que ce sur quoi il a été entraîné. Tes documents internes, il ne les a jamais vus.

TempsCe qui se passeRésultat
1. Retrieval
(récupération)		 On découpe le document en petits morceaux. Quand tu poses une question, le système cherche les morceaux qui parlent du même sujet que ta question. On isole les 15 passages les plus pertinents sur 100+ morceaux au total.
2. Generation
(génération)		 On donne ces 15 passages au LLM avec ta question. Le LLM lit uniquement ces passages et rédige sa réponse en citant d'où vient chaque information. Réponse précise, fondée sur le document, avec page et paragraphe. Si l'info n'est pas dans les passages → refus explicite.
Pourquoi ça élimine l'hallucination

Le LLM ne répond jamais de mémoire. Il a toujours le texte source sous les yeux. Pas de texte source = pas de réponse.

HTTPS = HTTP + chiffrement. HTTP est le protocole standard pour que deux machines communiquent sur Internet (le navigateur demande, le serveur répond). Le « S » signifie que les données sont chiffrées pendant le transport — personne ne peut les lire en chemin. Analogie : HTTP = carte postale (lisible par tous), HTTPS = lettre scellée (seul le destinataire peut l'ouvrir). Render fournit automatiquement ce chiffrement.

Le flux : le frontend n'a aucune logique métier. Il envoie des requêtes HTTPS au backend, qui fait tout le travail (parsing, chunking, embedding, search, generation) et retourne du JSON (un format texte structuré que les machines savent lire).

numpy store : ce sont 2 variables Python stockées dans la mémoire vive (RAM) du serveur — la liste des morceaux de texte et leurs vecteurs associés. Ce n'est pas une base de données — si le serveur redémarre, tout est perdu et l'utilisateur doit ré-uploader son document.

5

Le pipeline — 5 étapes

Chaque étape est un module Python indépendant. Aucun module ne dépend d'un autre — ils sont tous orchestrés par api.py (le chef d'orchestre).

2 flux distincts — ne pas confondre

Upload (l'utilisateur envoie un fichier) : le frontend envoie le fichier au backend, qui exécute PARSE → CHUNK → EMBED (appel à OpenAI) → STORE. Le backend répond « prêt, 48 morceaux indexés ». Pas de réponse générée ici — on prépare le terrain.

Query (l'utilisateur pose une question) : le backend transforme la question en vecteur (appel OpenAI), cherche les 15 morceaux les plus proches (numpy, <1ms), puis envoie ces morceaux + la question à GPT-4o-mini (2e appel OpenAI) qui rédige la réponse avec citations.

Le frontend ne parle jamais à OpenAI directement. Tout passe par le backend. Le frontend ne sait même pas qu'OpenAI existe.

1
PARSE extraction · rag/parser.py
Ce qui se passe

On extrait le texte brut du fichier.

PDF → pdfplumber, page par page. On enregistre les positions de chaque page dans le texte (page 1 = caractères 0 à 2400, page 2 = 2401 à 4800...).

CSV → chaque ligne convertie en phrase lisible.

TXT → lecture directe.

Pourquoi on fait ça

L'enregistrement des positions de pages est essentiel pour les citations.

C'est grâce à ça qu'à la fin, quand le système donne une réponse, il peut remonter le fil et dire « cette information vient de la Page 22 ».

Sans ça → réponse correcte mais aucun moyen de savoir d'où elle vient dans le document.

2
CHUNK découpage · rag/chunker.py
Ce qui se passe

Le texte est découpé en blocs de 500 tokens (~375 mots). Chaque bloc chevauche le suivant de 100 tokens.

Tokenizer : cl100k_base (tiktoken — le même tokenizer utilisé par les modèles OpenAI).

Pourquoi on fait ça

La recherche est plus précise sur des petits blocs ciblés que sur des pages entières.

Le chevauchement de 100 tokens sert à ne jamais couper une idée en deux : si une phrase est à la frontière entre deux blocs, elle apparaît dans les deux.

3
EMBED vectorisation · rag/embedder.py
Ce qui se passe

Chaque chunk est envoyé via Internet (HTTPS) au modèle text-embedding-3-small (hébergé chez OpenAI) par batchs de 50. OpenAI retourne un vecteur pour chaque chunk — c'est un aller-retour réseau, pas un calcul local.

Pourquoi on fait ça

C'est cette étape qui rend la recherche par sens possible — chercher par signification au lieu de chercher par mots-clés.

C'est quoi un vecteur, concrètement ?

Un vecteur, c'est une liste ordonnée de nombres. Ici, chaque chunk est transformé en une liste de 1536 nombres décimaux — par exemple : [0.023, -0.041, 0.089, ..., 0.012].

Pourquoi 1536 ? C'est la taille fixée par le modèle d'embedding utilisé. Chaque nombre représente une dimension du sens. Comme des coordonnées GPS : au lieu de 2 dimensions (latitude, longitude), on a 1536 dimensions pour capturer toutes les nuances de sens d'un texte.

Ce que ça permet : « coût d'acquisition client » et « combien on dépense pour acquérir un utilisateur » → vecteurs très proches, même si aucun mot n'est commun.

4
SEARCH recherche · rag/store.py
Ce qui se passe

La question est aussi transformée en vecteur (via le même modèle d'embedding, 2e appel à OpenAI). Ensuite, numpy compare ce vecteur avec tous les vecteurs des chunks grâce à la similarité cosinus.

Analogie : imagine que chaque morceau de texte est une flèche qui pointe dans une direction. La similarité cosinus mesure si deux flèches pointent dans la même direction (= parlent du même sujet). 1.0 = même direction exacte. 0 = rien à voir.

Concrètement : numpy fait un calcul mathématique (produit scalaire normalisé) entre le vecteur de ta question et chacun des ~240 vecteurs stockés. Ça prend moins d'1 milliseconde. On trie les résultats et on garde les 15 chunks les plus proches.

Pourquoi on fait ça

C'est le cœur du « Retrieval » dans RAG. Sur 100+ chunks, on isole les 15 qui parlent du même sujet que ta question.

Pourquoi 15 chunks ?

Ce n'est pas un standard — c'est un paramètre (TOP_K) qu'on ajuste. J'ai commencé à 5, puis 10, puis 15. À 10, une question factuelle échouait : le chunk qui contenait le bon chiffre n'était pas récupéré.

En entreprise, ce paramètre dépend de la taille des documents et du coût. On ajoute souvent une étape de re-ranking (un deuxième modèle qui reclasse les chunks par pertinence).

5
GENERATE réponse · rag/generator.py
Ce qui se passe

Les 15 chunks sont envoyés à GPT-4o-mini (un modèle différent de celui qui fait les embeddings) avec la question. C'est un 2e appel HTTPS à OpenAI. Le LLM a un system prompt strict (des instructions qu'il doit suivre) : répondre uniquement depuis le texte fourni, citer la source pour chaque fait, refuser si l'info n'y est pas.

Pourquoi on fait ça

C'est le « Generation » dans RAG. Le LLM ne répond jamais de mémoire — il a toujours les passages source sous les yeux.

Comment marchent les citations

Avant d'envoyer les chunks au LLM, on annote chaque paragraphe avec un marqueur : le 1er paragraphe reçoit [P1] (= Paragraphe 1), le 2ème [P2], etc.

Résultat : le LLM peut citer « Page 22, Paragraphe 3 » — pas juste « Page 22 ».

Le system prompt contient aussi une règle anti-jargon : le LLM n'a pas le droit d'utiliser des termes techniques (chunks, embeddings, retrieval) dans ses réponses. L'utilisateur final voit « le document », pas les rouages du pipeline.

Les 3 endpoints de l'API

EndpointMéthodeCe qu'il reçoitCe qu'il faitCe qu'il retourne
/health GET Rien Vérifie que le serveur tourne {"status": "ok"}
/upload POST Un fichier (PDF, TXT ou CSV) Vérifie taille ≤10 MB → parse → chunk → embed → stocke en mémoire {"num_chunks": 48, "status": "ready"}
/query POST {"question": "..."} Embed la question → cherche 15 chunks → envoie au LLM {"answer": "...", "latency": 2.3, "sources": [...]}

Ce que sources contient : pour chaque chunk utilisé — son index, la page de début/fin, la distance cosinus, et les 150 premiers caractères du texte. C'est ce qui permet au frontend d'afficher les citations.

Le parcours d'un fichier — de l'upload à « ready »

Utilisateur clique « Upload » dans le navigateur
        ↓
POST /upload (le fichier part via le réseau)
        ↓
Vérification taille : > 10 MB ? → HTTP 413 « File too large »
        ↓
Adapter : traduit le format du fichier reçu vers le format attendu par le parser
        ↓
PARSE → CHUNK → EMBED → STORE (même pipeline que les 4 premières étapes)
        ↓
Réponse : {"num_chunks": 48, "status": "ready"}

Single-tenant : le backend ne garde qu'un seul document en mémoire à la fois. Si un autre utilisateur upload un fichier, le précédent est effacé. Si le serveur redémarre, tout est perdu. C'est un choix de prototype : pas de base de données, pas de sessions.

9

CORS — le problème cross-origin

C'est quoi : quand ton frontend (docuqueryai.lovable.app) appelle ton backend (onrender.com), le navigateur bloque la requête par défaut. C'est une protection — un site web ne peut pas appeler n'importe quel serveur sans permission. CORS est le mécanisme qui dit au navigateur : « ce frontend a le droit de m'appeler ».

Le problème rencontré : Lovable a deux domaines — *.lovableproject.com (pendant le dev) et *.lovable.app (l'app publiée). La config initiale ne couvrait qu'un seul → ça marchait en dev mais pas en production.

La solution : une regex qui couvre les deux domaines d'un coup, plus les ports localhost pour le dev local.

Métriques d'évaluation

Un produit RAG peut échouer de 4 façons : réponse fausse, fait inventé, citation inexacte, ou réponse incohérente d'une fois à l'autre. Chaque métrique cible un de ces risques.

C'est quoi un « golden dataset » ?

C'est un ensemble fixe de 12 questions + réponses attendues, préparées à l'avance. On pose toujours les mêmes 12 questions à chaque version du produit, comme un examen standardisé. Pourquoi ? Parce que si tu poses des questions différentes à chaque test, tu ne peux pas savoir si une amélioration est réelle ou si la question était juste plus facile. Le golden dataset garantit que la comparaison est fiable.

Composition : 4 questions factuelles (un chiffre précis), 3 multi-hop (synthèse de plusieurs pages), 2 de synthèse (résumé), 2 adversariales (l'info n'existe PAS dans le document — le système doit refuser), 1 de consistance (re-poser la même question pour vérifier que la réponse ne change pas).

Chaque métrique est classée en 3 niveaux de sévérité :

NiveauSignificationInspiration
BLOCKINGSi ça échoue, on ne shippe pas. Non-négociable.Cooper Stage-Gate — certains critères ne se négocient pas
QUALITYSeuil configurable. Le Builder décide si c'est acceptable.RAGAS (framework d'éval RAG open-source, seuil usuel ~80%)
SIGNALOn mesure pour surveiller, mais ça ne bloque pas le lancement.Google SRE Error Budget — les causes externes sont monitorées, pas bloquantes

Le NO-GO → CONDITIONAL GO — ce qui s'est réellement passé

Au 1er passage de l'eval, la précision factuelle (G2) était à 75% (3/4). Mon seuil BLOCKING était à 100%. Résultat : NO-GO — on ne shippe pas.

Le diagnostic : une question demandait « combien de transactions par seconde au lancement ?». La bonne réponse (« 10,000 TPS », page 13) n'était pas dans les 10 morceaux récupérés. Le système trouvait un autre morceau (page 45, test de charge « 1,000 TPS ») qui était plus proche sémantiquement de « au lancement ».

Le fix : passer le nombre de morceaux récupérés (TOP_K) de 10 à 15. 1 ligne de code, 1 paramètre.

2e passage : G2 = 100% (4/4). Zéro régression sur les autres questions. → CONDITIONAL GO (0 BLOCKING fail, 1 SIGNAL fail sur la latence).

La leçon : sans évaluation rigoureuse, ce bug serait passé inaperçu. Le produit aurait donné des réponses fausses sans que personne ne le sache.

Concepts — si on te demande

ConceptC'est quoiExemple concret
Fenêtre de contexte Quantité max de texte qu'un LLM lit en une fois GPT-4 = 128K tokens, Claude = 200K tokens
Lost in the middle Le LLM traite bien le début et la fin, rate l'info au milieu du texte Ton chiffre clé est page 35/60 → risque qu'il passe à côté
Hallucination Le LLM invente des faits plausibles quand il n'a pas l'info « Le budget est de 12M » — chiffre qui n'existe nulle part
Embedding Liste de 1536 nombres qui capture le sens d'un texte. Comme des coordonnées GPS en 1536 dimensions. « coût d'acquisition » ≈ « combien pour acquérir un client »
Cosine similarity Mesure si 2 vecteurs pointent dans la même direction (= même sujet) 1.0 = même sens, 0 = rien à voir
Lexical vs sémantique Lexical = cherche les mots exacts. Sémantique = comprend le sens. Ctrl+F = lexical. Embedding search = sémantique.
TOP_K Nombre de chunks récupérés par la recherche. Paramètre ajustable, pas un standard. DocuQuery = 15. En entreprise → dépend du corpus.
Re-ranking Deuxième modèle qui reclasse les chunks par pertinence après la recherche initiale Utilisé en production enterprise pour optimiser TOP_K
FastAPI Framework Python pour créer des APIs. Tu définis des routes (« quand quelqu'un appelle /upload, fais ça »), FastAPI gère la validation, les erreurs, et la doc auto. @app.post("/upload") crée une route qui accepte des fichiers
Uvicorn Le serveur qui fait tourner FastAPI. FastAPI définit le comportement (« si on appelle /upload, fais ça »), mais il ne sait pas écouter Internet tout seul. Uvicorn est le programme qui ouvre un port réseau, reçoit les requêtes HTTP, et les transmet à FastAPI. C'est comme un standardiste téléphonique : il décroche les appels et les passe au bon service. Sans lui, FastAPI existe mais personne ne peut le contacter. uvicorn api:app --port 8000 = « écoute le port 8000 et passe tout à FastAPI »
Render Plateforme cloud (comme Heroku). Tu push du code sur GitHub, Render le détecte, l'installe et le met en ligne automatiquement. Push sur main → backend live en 2 minutes
Lovable Outil no-code qui génère du React + Tailwind via des prompts en langage naturel « Ajoute un stepper qui dit Reading your document » → code React généré
CORS Protection navigateur : un site web ne peut pas appeler un serveur sur un autre domaine sans permission. CORS est la liste blanche qui autorise certains frontends à appeler le backend. lovable.app veut appeler onrender.com → le backend doit l'autoriser
Adapter pattern Un traducteur entre deux formats incompatibles. Les 5 modules RAG ont été écrits pour Streamlit (le premier frontend local). Quand on est passé à FastAPI, le format des fichiers uploadés a changé. Au lieu de réécrire les 5 modules, on a créé un traducteur qui convertit le nouveau format vers l'ancien. Résultat : 0 changement dans les modules RAG. FastAPI envoie un UploadFile, les modules attendent .name et .read(). L'adapter traduit l'un vers l'autre.
Multipart/form-data Le format standard pour envoyer un fichier via le web. C'est comme un email avec une pièce jointe : le navigateur envoie une requête qui contient à la fois des infos (le nom du fichier) et les données brutes du fichier. C'est ce format que /upload attend. Tu cliques « Upload » → le navigateur emballe le PDF et l'envoie au serveur
Single-tenant Un seul utilisateur/document à la fois. Pas de sessions, pas de multi-utilisateurs. Choix de prototype. Upload = efface le document précédent
Stateless Le serveur ne persiste rien sur disque. Redémarrage = données perdues. Render restart → l'utilisateur doit ré-uploader son document

Réponses — vérifie tes réponses

1
Dessine le schéma d'architecture

3 couches : Frontend (React/Tailwind sur Lovable) → Backend (FastAPI sur Render) → OpenAI API. Plus le numpy store en dessous du backend.

Communication : Frontend ↔ Backend en HTTPS (requêtes + réponses JSON). Backend ↔ OpenAI en HTTPS (embeddings + génération). Le frontend ne parle jamais à OpenAI directement. numpy store = 2 variables Python en RAM, reliées au backend en interne (pas de réseau).

2
Les 3 endpoints de l'API

GET /health → reçoit rien, retourne {"status": "ok"}. Vérifie que le serveur tourne.

POST /upload → reçoit un fichier (PDF, TXT ou CSV, max 10 MB), exécute parse → chunk → embed → store, retourne {"num_chunks": 48, "status": "ready"}.

POST /query → reçoit {"question": "..."}, embed la question → cherche 15 chunks → envoie au LLM, retourne {"answer": "...", "latency": 2.3, "sources": [...]}.

3
C'est quoi CORS ? Pourquoi c'était un problème avec Lovable ?

CORS = protection du navigateur. Par défaut, un site web ne peut pas appeler un serveur sur un autre domaine. CORS est la liste blanche côté backend qui dit « ce frontend a le droit de m'appeler ».

Le problème Lovable : Lovable utilise 2 domaines différents — *.lovableproject.com en dev et *.lovable.app en production. La config initiale ne couvrait qu'un seul → ça marchait en dev mais cassait en prod. Fix : une regex qui couvre les deux domaines d'un coup.

4
Parcours complet d'un PDF de l'upload à « ready »

L'utilisateur clique Upload → le navigateur emballe le PDF en multipart/form-data et l'envoie en POST à /upload → le backend vérifie la taille (>10 MB = rejeté, HTTP 413) → l'adapter traduit le format FastAPI vers le format attendu par le parser → PARSE : pdfplumber extrait le texte page par page et enregistre les positions de chaque page → CHUNK : découpage en blocs de 500 tokens avec 100 tokens de chevauchement → EMBED : chaque chunk est envoyé à OpenAI (text-embedding-3-small) par batchs de 50, qui retourne des vecteurs de 1536 nombres → STORE : les chunks + vecteurs sont stockés en RAM (numpy) → réponse JSON {"status": "ready"}.

5
Pourquoi single-tenant ?

Le backend ne garde qu'un seul document en mémoire à la fois. Si quelqu'un d'autre upload un fichier, le précédent est effacé. Si le serveur redémarre, tout est perdu.

Pourquoi ce choix : c'est un prototype. Pas de base de données, pas de sessions, pas de multi-utilisateurs. Ça simplifie tout — zéro gestion d'état persistant. Le but est de prouver que le pipeline RAG fonctionne, pas de gérer 10,000 utilisateurs.

6
Pourquoi ChromaDB remplacé par numpy ?

Le problème : ChromaDB utilise SQLite en interne. SQLite ne supporte pas les dossiers synchronisés iCloud (conflit de verrous fichiers). Résultat : crash au démarrage.

La solution : numpy — ~50 lignes de code. 2 variables en RAM (les chunks + leurs vecteurs). La recherche est un produit scalaire normalisé (similarité cosinus) calculé en brute force. Pour ~240 chunks (1 document), ça prend <1ms. Pas besoin d'index optimisé à cette échelle.

7
C'est quoi l'adapter pattern dans api.py ?

Les 5 modules RAG ont été écrits pour Streamlit (le premier frontend local). Streamlit envoie un objet avec .name et .read(). Quand on est passé à FastAPI, le format a changé (.filename au lieu de .name, méthodes async).

Au lieu de réécrire les 5 modules, on a créé un « traducteur » (_UploadFileAdapter) qui convertit le format FastAPI vers le format attendu. Résultat : 0 changement dans les modules RAG.

8
C'est quoi Uvicorn ?

FastAPI définit le comportement (« si on appelle /upload, fais ça »), mais il ne sait pas écouter Internet tout seul. Uvicorn est le programme qui ouvre un port réseau, reçoit les requêtes HTTP, et les transmet à FastAPI.

Analogie : Uvicorn = standardiste téléphonique. Il décroche les appels et les passe au bon service (FastAPI). Sans lui, FastAPI existe mais personne ne peut le contacter.

9
Pourquoi Python 3.11 ?

tiktoken (le tokenizer qui découpe le texte en tokens) ne supportait pas Python 3.14 au moment du build. Pin explicite à 3.11 dans la config Render pour éviter le problème.

10
Le pipeline RAG en 5 étapes

PARSE : extraire le texte brut du fichier (PDF page par page, CSV en phrases, TXT direct). On enregistre où commence chaque page pour les citations.

CHUNK : découper en blocs de 500 tokens (~375 mots) avec 100 tokens de chevauchement pour ne jamais couper une idée en deux.

EMBED : envoyer chaque bloc à OpenAI qui le transforme en vecteur (liste de 1536 nombres capturant le sens). C'est un appel réseau, pas un calcul local.

SEARCH : transformer la question en vecteur, comparer avec tous les vecteurs stockés (similarité cosinus), garder les 15 plus proches. Calcul numpy en <1ms.

GENERATE : envoyer les 15 blocs + la question à GPT-4o-mini (modèle différent de celui des embeddings). Le LLM répond uniquement depuis ces passages, avec citations page + paragraphe.

11
C'est quoi un embedding ? Pourquoi 1536 ?

Un embedding, c'est une liste ordonnée de nombres décimaux qui capture le sens d'un texte. Exemple : [0.023, -0.041, 0.089, ..., 0.012].

Analogie : comme des coordonnées GPS, mais au lieu de 2 dimensions (latitude, longitude), on a 1536 dimensions pour capturer toutes les nuances de sens. Deux textes qui parlent du même sujet auront des coordonnées proches, même s'ils n'utilisent aucun mot en commun.

Pourquoi 1536 : c'est la taille fixée par le modèle text-embedding-3-small d'OpenAI. On ne choisit pas ce nombre — c'est le modèle qui décide combien de dimensions il a besoin pour représenter le sens.

12
Pourquoi TOP_K = 15 ?

Ce n'est PAS un standard. C'est un paramètre qu'on ajuste selon le projet. J'ai commencé à 5, puis 10, puis 15.

À 10, une question factuelle échouait : le chunk contenant le bon chiffre (page 13) n'était pas récupéré. Un chunk différent (page 45, test de charge) était sémantiquement plus proche de la question. Passer à 15 a résolu le problème — on ratisse plus large, donc plus de chances de capturer le bon passage.

Trade-off : plus de chunks = plus de contexte pour le LLM = légèrement plus cher et plus lent. Mais pour un seul document, 15 chunks c'est négligeable.

13
Comment fonctionnent les citations ?

Étape 1 (PARSE) : quand on extrait le texte d'un PDF, on enregistre où commence et finit chaque page (page 1 = caractères 0 à 2400, page 2 = 2401 à 4800...).

Étape 2 (CHUNK) : chaque chunk sait à quelles pages il appartient grâce à ces positions.

Étape 5 (GENERATE) : avant d'envoyer les chunks au LLM, on ajoute des marqueurs [P1], [P2], etc. au début de chaque paragraphe dans le chunk. Le LLM peut alors citer « Page 22, Paragraphe 3 » au lieu de juste « Page 22 ».

Le system prompt oblige aussi le LLM à citer chaque fait. S'il ne peut pas citer → il ne doit pas affirmer.

14
Flux Upload vs Query — appels OpenAI

Upload (1 appel OpenAI) : le fichier arrive au backend → PARSE → CHUNK → EMBED (appel à text-embedding-3-small pour transformer les chunks en vecteurs) → STORE en RAM. Pas de réponse générée, pas de GPT-4o-mini. Le backend répond « ready ».

Query (2 appels OpenAI) : la question arrive → appel 1 : embed la question (text-embedding-3-small) → SEARCH (numpy, local, <1ms) → appel 2 : envoie les 15 chunks + question à GPT-4o-mini qui rédige la réponse avec citations.

15
Pourquoi la précision factuelle est BLOCKING à 100% ?

Parce qu'une erreur sur un fait pur (un chiffre, un nom, une date) détruit la confiance dans le produit. Si le système dit « le budget est de 12M » alors que c'est 18.5M, l'utilisateur ne peut plus rien croire.

Les questions factuelles sont les plus simples — un chiffre précis qui existe dans le document. Si le système échoue là-dessus, il échouera partout. C'est le plancher, pas le plafond. D'où BLOCKING (non-négociable) et pas QUALITY (configurable).

16
C'est quoi G6 (Consistance) ?

On repose exactement la même question à deux moments différents (E1 et E12 dans le golden dataset). Les faits doivent être identiques — le wording peut varier, mais les chiffres et noms doivent être les mêmes.

Pourquoi c'est important : un système qui dit « $3,600 » puis « $4,200 » pour la même question est inutilisable en contexte professionnel. La consistance garantit que le système est fiable, pas juste chanceux.

Résultat DocuQuery : PASS — les deux réponses donnent $3,600 avec les mêmes détails (LTV:CAC 8.7:1).

17
Le NO-GO → CONDITIONAL GO en 30 secondes

« Premier eval : précision factuelle à 75%, mon seuil BLOCKING était à 100%. NO-GO. Une question demandait les TPS au lancement — le système trouvait 1,000 TPS (test de charge, page 45) au lieu de 10,000 TPS (spec fonctionnelle, page 13). Le chunk était sémantiquement plus proche de « au lancement » mais factuellement faux. Fix : TOP_K de 10 à 15, une seule ligne. Deuxième eval : 100% factuel, zéro régression. CONDITIONAL GO — le seul fail restant est la latence, une dépendance externe. »

18
C'est quoi un golden dataset ?

Un ensemble fixe de questions avec les réponses attendues, préparées à l'avance. On pose toujours les mêmes questions à chaque version du produit.

Pourquoi : si tu changes les questions à chaque test, tu ne sais pas si une amélioration vient du code ou de la difficulté des questions. Le golden dataset = un examen standardisé qui rend la comparaison fiable.

Composition DocuQuery (12 questions) : 4 factuelles (chiffre précis), 3 multi-hop (croiser plusieurs pages), 2 synthèse (résumer), 2 adversariales (l'info n'existe pas — le système doit refuser), 1 consistance (re-poser E1 pour vérifier).
